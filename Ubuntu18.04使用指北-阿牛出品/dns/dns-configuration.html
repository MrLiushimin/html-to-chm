组态

有很多方法可以配置BIND9。一些最常见的配置是缓存名称服务器，主要主服务器和辅助主服务器。

  • 当配置为缓存名称服务器时，BIND9将找到命名查询的答案，并在再次查询域时记住答案。

  • 作为主要主服务器，BIND9从其主机上的文件中读取区域的数据，并且对该区域具有权威性。

  • 在辅助主配置中，BIND9从该区域的另一个名称服务器获取区域数据。

概观

DNS配置文件存储在 / etc / bind目录中。主配置文件是 /etc/bind/named.conf。

在包括线指定包含DNS选项的文件名。/etc/bind/named.conf.options文件中的 目录行告诉DNS在哪里查找文件。BIND使用的所有文件都与该目录相关。

名为/etc/bind/db.root的文件 描述了世界上的根名称服务器。服务器随时间而变化，因此 必须立即维护/etc/bind/db.root文件。这通常作为bind9包的更新来完成。所述区部分定义一个主服务器，并且它被存储在在所提到的文件中的文件的选项。

可以将同一服务器配置为缓存名称服务器，主要主服务器和辅助主服务器。服务器可以是一个区域的授权开始（SOA），同时为另一个区域提供辅助服务。一直为本地LAN上的主机提供缓存服务。

缓存Nameserver

默认配置设置为充当缓存服务器。只需添加ISP的DNS服务器的IP地址即可。只需在/etc/bind/named.conf.options中取消注释并编辑以下内容： 

货代{
                1.2.3.4;
                5.6.7.8;
           };

将1.2.3.4和5.6.7.8替换为实际名称服务器的IP地址。

现在重新启动DNS服务器，以启用新配置。从终端提示： 

sudo systemctl restart bind9.service

有关测试缓存DNS服务器的信息，请参阅dig。

小学硕士

在本节中，BIND9将被配置为域example.com的主要主服务器 。只需将example.com替换为您的FQDN（完全限定域名）即可。

转发区文件

要向BIND9添加DNS区域，将BIND9转换为主要主服务器，第一步是编辑 /etc/bind/named.conf.local： 

zone“example.com”{
	类型大师;
        文件“/etc/bind/db.example.com”;
};

（注意，如果bind将像DDNS一样接收文件的自动更新，那么在此处和复制命令中使用/var/lib/bind/db.example.com 而不是/etc/bind/db.example.com下面。）

现在使用现有区域文件作为模板来创建/etc/bind/db.example.com文件： 

sudo cp /etc/bind/db.local /etc/bind/db.example.com

编辑新区域文件/etc/bind/db.example.com更改localhost。 到服务器的FQDN，留下额外的“。” 在末尾。将127.0.0.1更改为名称服务器的IP地址，将root.localhost更改为有效的电子邮件地址，但使用“。” 而不是通常的“@”符号，再次离开“。” 在末尾。更改注释以指示此文件所针对的域。

为基本域example.com 创建A记录。此外，为ns.example.com创建A记录，在此示例中为名称服务器： 

;
; example.com的BIND数据文件
;
$ TTL 604800
@ IN SOA example.com。root.example.com。（
                              2; 串行
                         604800; 刷新
                          86400; 重试
                        2419200; 到期
                         604800）; 负缓存TTL
        在A 192.168.1.10
;
@ IN NS ns.example.com。
@ IN A 192.168.1.10
@ IN AAAA :: 1
ns IN A 192.168.1.10

每次更改区域文件时，都 必须增加序列号。如果在重新启动BIND9之前进行多项更改，只需将Serial增加一次即可。

现在，您可以将DNS记录添加到区域文件的底部。有关详细信息， 请参阅常用记录

许多管理员喜欢使用编辑为区域序列的最后日期，例如 2012010100 ，即yyyymmddss（其中ss是序列号）

对区域文件进行更改后，需要重新启动BIND9才能使更改生效： 

sudo systemctl restart bind9.service

反向区域文件

既然区域已设置并将名称解析为IP地址，则还需要反向区域。反向区域允许DNS将地址解析为名称。

编辑/etc/bind/named.conf.local并添加以下内容： 

zone“1.168.192.in-addr.arpa”{
        类型大师;
        文件“/etc/bind/db.192”;
};

将1.168.192替换为您正在使用的任何网络的前三个八位字节。另外，适当地命名区域文件/etc/bind/db.192。它应该匹配您的网络的第一个八位字节。

现在创建/etc/bind/db.192文件： 

sudo cp /etc/bind/db.127 /etc/bind/db.192

接下来编辑/etc/bind/db.192改变基本相同的选项 /etc/bind/db.example.com： 

;
; BIND本地192.168.1.XXX网络的反向数据文件
;
$ TTL 604800
@ IN SOA ns.example.com。root.example.com。（
                              2; 串行
                         604800; 刷新
                          86400; 重试
                        2419200; 到期
                         604800）; 负缓存TTL
;
@ IN NS ns。
10 IN PTR ns.example.com。

反向区域中 的序列号也需要在每次更改时递增。对于在/etc/bind/db.example.com中配置的每个A记录，即不同的地址，您需要在/etc/bind/db.192中创建PTR记录。

创建反向区域文件后重启BIND9： 

sudo systemctl restart bind9.service

中学硕士

一旦主主已经配置了二级硕士是必要的，以保持域名的可用性，如果主变得不可用。

首先，在Primary Master服务器上，需要允许区域传输。将allow-transfer 选项添加到/etc/bind/named.conf.local中的示例正向和反向区域定义： 

zone“example.com”{
        类型大师;
	文件“/etc/bind/db.example.com”;
        允许转移{192.168.1.11; };
};

zone“1.168.192.in-addr.arpa”{
        类型大师;
        文件“/etc/bind/db.192”;
	允许转移{192.168.1.11; };
};

将192.168.1.11替换为辅助名称服务器的IP地址。

在Primary Master上 重新启动BIND9： 

sudo systemctl restart bind9.service

接下来，在辅助主服务器上，以与主服务器上相同的方式安装bind9软件包。然后编辑/etc/bind/named.conf.local并为Forward和Reverse区域添加以下声明： 

zone“example.com”{
	类型奴隶;
        文件“db.example.com”;
        大师{192.168.1.10; };
};        
      
zone“1.168.192.in-addr.arpa”{
	类型奴隶;
        档案“db.192”;
        大师{192.168.1.10; };
};

将192.168.1.10替换为主名称服务器的IP地址。

在辅助主服务器上 重新启动BIND9： 

sudo systemctl restart bind9.service

在/ var / log / syslog中，您应该看到类似的内容（某些行已被拆分以适合此文档的格式）：

注意：仅当主节点上的序列号大于辅助节点上的序列号时，才会传送区域。如果您希望让主要主DNS通知备用DNS服务器区域更改，您可以添加- 通知{ipaddress; }; 在/etc/bind/named.conf.local中，如下例所示： 

zone“example.com”{
	类型大师;
	文件“/etc/bind/db.example.com”;
	允许转移{192.168.1.11; };
	还 - 通知{192.168.1.11; };
	};

zone“1.168.192.in-addr.arpa”{
	类型大师;
	文件“/etc/bind/db.192”;
	允许转移{192.168.1.11; };
	还 - 通知{192.168.1.11; };
	};

非权威区域文件的缺省目录是/ var / cache / bind /。此目录也在AppArmor中配置，以允许named守护程序写入它。有关AppArmor的更多信息，请参阅AppArmor。
