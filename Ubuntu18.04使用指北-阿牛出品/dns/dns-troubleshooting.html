故障排除

本节介绍了在DNS和BIND9出现问题时帮助确定原因的方法。

  • 测试
  • 记录

测试

的resolv.conf

测试BIND9的第一步是将名称服务器的IP地址添加到主机解析器。应配置主名称服务器以及另一个主机以进行双重检查。有关向网络客户端添加名称服务器地址的详细信息，请参阅DNS客户端配置，然后检查文件/etc/resolv.conf是否包含（对于此示例）： 

名称服务器192.168.1.10
nameserver 192.168.1.11

在127. *监听的名称服务器负责将自己的IP地址添加到resolv.conf（使用resolvconf）。这是通过文件/ etc / default / bind9将RESOLVCONF = no更改为RESOLVCONF = yes来完成的。

如果主节点不可用，您还应该添加辅助名称服务器的IP地址。

挖

如果您安装了dnsutils软件包，则可以使用DNS查找实用程序挖掘来测试您的设置 ：

  • 安装BIND9后，使用dig对回环接口，以确保它正在侦听端口53.从终端提示符： 

    dig -x 127.0.0.1

    您应该在命令输出中看到类似于以下内容的行： 

    ;; 查询时间：1毫秒
;; 服务器：192.168.1.10＃53（192.168.1.10）

  • 如果您已将BIND9配置为缓存名称服务器，则“挖掘”外部域以检查查询时间： 

    挖掘ubuntu.com

    请注意命令输出结束时的查询时间： 

    ;; 查询时间：49毫秒

    经过第二次挖掘后，应该有所改进： 

    ;; 查询时间：1毫秒

平

现在，为了演示应用程序如何使用DNS来解析主机名，请使用 ping实用程序发送ICMP回送请求。从终端提示输入： 

ping example.com

这将测试名称服务器是否可以将名称ns.example.com解析为IP地址。命令输出应类似于： 

PING ns.example.com（192.168.1.10）56（84）字节的数据。
来自192.168.1.10的64字节：icmp_seq = 1 ttl = 64时间= 0.800毫秒
来自192.168.1.10的64字节：icmp_seq = 2 ttl = 64时间= 0.813 ms

命名checkzone

测试区域文件的一种好方法是使用随bind9包安装的named-checkzone实用 程序。此实用程序允许您在重新启动BIND9并进行实时更改之前确保配置正确。

  • 要测试我们的示例前向区域文件，请从命令提示符输入以下内容： 

    named-checkzone example.com /etc/bind/db.example.com

    如果一切配置正确，您应该看到类似于的输出： 

    zone example.com/IN：已加载序列号6
好

  • 同样，要测试反向区域文件，请输入以下内容： 

    named-checkzone 1.168.192.in-addr.arpa /etc/bind/db.192

    输出应类似于： 

    区域1.168.192.in-addr.arpa/IN：已加载序列号3
好

您的区域文件 的序列号可能会有所不同。

记录

BIND9具有多种日志配置选项。有两个主要选择。该 通道选项配置，其中原木去，和类别选项决定信息登录。

如果未配置日志记录选项，则默认选项为： 

记录{
     category default {default_syslog; default_debug; };
     category unmatched {null; };
};

本节介绍如何配置BIND9以将与DNS查询相关的调试消息发送到单独的文件。

  • 首先，我们需要配置一个通道来指定将消息发送到哪个文件。编辑/etc/bind/named.conf.local 并添加以下内容： 

    记录{
    channel query.log {      
        文件“/var/log/query.log”;
        严重性调试3; 
    }; 
};

  • 接下来，配置一个类别以将所有DNS查询发送到查询文件： 

    记录{
    channel query.log {      
        文件“/var/log/query.log”; 
        严重性调试3; 
    }; 
    类别查询{query.log; }; 
};

注意：debug选项可以设置为1到3.如果未指定级别，则级别1是默认值。

  • 由于named守护程序作为绑定用户运行，因此 必须创建/var/log/query.log文件并更改所有权： 

    sudo touch /var/log/query.log 
sudo chown bind /var/log/query.log

  • 在named守护程序可以写入新日志文件之前， 必须更新AppArmor配置文件。首先，编辑/etc/apparmor.d/usr.sbin.named并添加： 

    /var/log/query.log w，

    接下来，重新加载配置文件： 

    cat /etc/apparmor.d/usr.sbin.named | sudo apparmor_parser -r

    有关AppArmor的更多信息，请参阅AppArmor

  • 现在重新启动BIND9以使更改生效： 

    sudo systemctl restart bind9.service

您应该看到文件/var/log/query.log填充查询信息。这是 BIND9日志记录选项的一个简单示例。有关高级选项的介绍，请参阅更多信息。
